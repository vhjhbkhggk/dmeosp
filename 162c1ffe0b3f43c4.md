# MetaRepo Index

MetaRepo 是一个面向技术调研与知识管理场景的轻量级外链元数据归档系统。项目定位为技术资源导航与结构化引用仓库，帮助开发者、技术作者与团队知识管理员将散落在代码托管平台、技术文档与内部 Wiki 中的外部参考链接统一收录，并通过规范的 Markdown 索引体系进行版本化管理。

项目目标用户包括需要长期维护技术选型参考库的架构师、撰写技术博客时需批量管理引用来源的内容创作者，以及希望将外部技术资源与内部研发流程形成稳定映射的 DevOps 团队。MetaRepo 不提供爬虫或自动抓取能力，而是围绕人工筛选与结构化标注的工作流，提供清晰的目录模板、索引约定与批量更新脚本，使外链资源在项目演进中保持可追溯、可审计、可补全的状态。

## 功能概览

**批量链接收录**：支持通过脚本将零散的外部链接按既定命名规范导入 `/references` 目录，每条链接自动生成带时间戳与哈希指纹的占位 Markdown 文件。

**元数据标注框架**：为每条链接提供统一的 frontmatter 区块，包含资源类型、技术领域、收录理由、关键标签、关联内部项目编号与最后复核日期。

**标签化分类视图**：基于标签系统自动生成多维度分类索引，支持按编程语言、框架版本、基础设施组件或业务领域进行动态筛选。

**变更跟踪与审核日志**：每次新增或更新外链时，强制要求填写变更说明，并将操作记录写入 `/changelog` 目录，形成完整的资源库审计轨迹。

**静态站点生成适配**：索引数据与 Markdown 源文件兼容主流静态站点生成器，可一键导出为可浏览的 HTML 帮助中心或团队知识门户。

**冲突检测与去重提醒**：在收录新链接时自动比对已有资源库，若检测到相同 URL 或高度相似的标题，给出冲突警告并建议合并或替换。

**批量导出为 CSV 或 JSON**：支持将整个索引库导出为结构化数据格式，便于导入外部协作平台或进行二次统计分析与可视化。

## 应用场景

技术选型调研过程中的参考链接归档。当团队针对某一中间件或云服务进行横向对比时，可将各候选产品的官方文档、性能评测文章、社区案例与安全公告统一收录至 MetaRepo，形成可复用的调研证据包。

技术博客与外部内容引用管理。技术作者在撰写系列文章时，可将参考的外部资料、代码片段出处与数据来源集中存放于 MetaRepo 的 `/citations` 路径下，确保每篇博文的引用信息准确且便于后期修订。

内部知识库与外部资源的双向映射。企业研发中心可将供应商提供的私有文档、补丁说明与公开 GitHub 仓库中的相关 Issue 讨论建立关联，通过 MetaRepo 的 `related` 字段维护内部工单系统与外部链接的对应关系。

开源项目外部依赖溯源。开源项目维护者可使用 MetaRepo 记录依赖库的官方网站、迁移公告、CVE 修复记录与替代方案讨论帖，在版本升级或安全审计时快速定位关键外部信息源。

## 快速开始

以下命令演示如何在本地环境克隆 MetaRepo 索引库、安装基础依赖并启动本地预览服务。

```bash
git clone https://github.com/your-org/metarepo-index.git
cd metarepo-index
npm install
npm run build
npm run serve
```

若使用 Python 环境，可选用以下方式：

```bash
git clone https://github.com/your-org/metarepo-index.git
cd metarepo-index
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python scripts/ingest.py --input ./new_links.csv
```

执行上述命令后，本地服务默认启动于 `http://localhost:8080`，可访问索引首页查看已收录的资源列表与分类导航。

## 贡献指南

提交新链接或更新现有索引时，请遵循以下流程以保证索引库的一致性与可维护性。

第一，克隆项目仓库并在本地新建功能分支，分支命名采用 `feat/add-resource-{批次号}` 或 `fix/update-metadata-{链接哈希}` 格式，避免直接在 main 分支操作。

第二，新增链接时使用 `scripts/ingest.js` 脚本，通过 `--input` 参数指定包含 URL 列表的 CSV 或纯文本文件，脚本自动生成符合 frontmatter 模板的 Markdown 文件并存放至 `references/` 下相应子目录。若手动创建索引文件，必须完整填写 `title`、`url`、`tags`、`category`、`description` 与 `reviewed_at` 字段。

第三，更新已有索引时，需在 frontmatter 中递增 `version` 字段并在 `changelog` 数组中追加一条变更记录，说明修改原因与影响范围。所有变更需同步更新 `/changelog` 目录下当日的汇总文件。

第四，提交前运行 `npm run test` 与 `npm run lint` 确保脚本通过单元测试且所有 Markdown 文件符合格式规范。CI 流水线将自动拦截格式错误或 frontmatter 缺失的提交。

第五，发起 Pull Request 并至少获得一位维护者的审阅。审阅者将检查链接是否重复、分类是否准确、标签是否在白名单内，以及描述是否清晰。合并后索引库自动重新构建并更新静态预览站点。

## 常见问题

Q: 索引库中已存在某条链接，但我想更新其描述或标签，应该如何操作而不触发冲突警告？

A: 直接修改对应 Markdown 文件的 frontmatter 部分，同时递增 `version` 字段并在 `changelog` 数组中添加一条记录，说明本次变更内容。提交前运行 `scripts/dedup.js` 确认没有与其他条目产生新的冲突。若因链接变更导致与其他条目标题或描述高度相似，需根据脚本提示合并或调整标签以消除歧义。

Q: 收录脚本支持批量导入，但我的链接列表包含大量非技术类或已失效的 URL，如何过滤？

A: 可在 `config/ignore-url-patterns.txt` 中添加正则表达式模式，匹配的链接将被 `ingest.js` 自动跳过并输出警告日志。对于已失效的链接，建议在收录前先通过 `scripts/check-urls.js` 进行可达性预检，该脚本会返回每一条链接的 HTTP 状态码，便于人工决策是否纳入索引。

Q: 静态预览站点无法显示最新收录的链接，可能是什么原因？

A: 首先确认索引文件已成功合并至 main 分支，且 CI 流水线中的构建任务（通常为 `npm run build`）已通过。若构建通过但页面未更新，检查 `config/` 下的分类映射配置是否涵盖新链接所使用的 `category` 字段，以及 `tags-whitelist.json` 是否包含新增的标签。若本地构建正常但远程站点异常，请检查部署服务的缓存策略，必要时手动触发缓存刷新。
