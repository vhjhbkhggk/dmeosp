# MQSaqL Resource Aggregator

MQSaqL 是一个面向开发者与运维工程师的技术资源外链汇总工具，专注于收集、分类与快速检索分散在代码仓库、技术博客与社区讨论中的高价值外部链接。项目本身不存储任何实际内容，仅提供结构化的链接索引与轻量级元数据描述，帮助团队在内部文档体系之外建立一套可维护的外部知识索引层。

项目定位为技术团队的辅助知识库底座，适用于需要频繁查阅外部参考资料、跟踪依赖库更新公告或回溯历史技术决策依据的场景。MQSaqL 通过统一的条目格式与分类标签，将原本零散的书签、笔记与聊天记录中的链接转化为可共享、可版本化、可审核的纯文本资产，从而降低信息孤岛效应，提升协作效率。

## 功能概览

- **原始链接直存**：以 Markdown 列表形式逐条收录用户提交的完整 URL，保留协议、路径与查询参数，不做任何自动补全或格式转换，确保引用精确性。

- **批次与条目追踪**：内置批次编号（当前为第 100/114 批）与条目计数，支持按导入批次筛选新资源，便于周期性审核与过期链接清理。

- **单条目注释字段**：每条链接可附加简短注释（例如「官方迁移指南」「性能对比测试」「安全公告 2026-01」），用于快速识别链接用途，无需点击跳转。

- **按主题分类视图**：资源列表按技术领域或来源仓库进行二级分类，分类标题由项目维护者根据条目内容动态调整，当前版本覆盖包管理、运行时环境、数据存储、CI/CD 与监控告警五个大类。

- **链接状态占位标记**：为每一条目预留状态字段（有效/待验证/失效），当前版本默认标记为「待验证」，后续可通过外部健康检查脚本批量更新。

- **全文检索支持**：生成的 Markdown 文档可直接配合 grep 或 ripgrep 进行本地关键词搜索，也可导入支持 Markdown 索引的文档工具（如 Obsidian、Logseq）建立双向链接。

- **纯文本可版本化**：所有资源定义以纯 Markdown 形式存在，完全兼容 Git 工作流，支持分支、合并与 Pull Request 审核，便于团队协作维护。

## 应用场景

**技术选型调研**：当团队需要评估多个中间件或框架时，可将候选项目的官网、GitHub 仓库、性能测试报告与社区对比文章的链接统一录入 MQSaqL，并添加各自适用场景的注释，形成一份可共享的调研对照表。

**故障排查知识沉淀**：在解决线上问题过程中，研发人员通常需要查阅大量外部文档（如官方 Issue 讨论、Stack Overflow 问答、第三方分析博客）。通过 MQSaqL 记录这些引用链接，并标注问题编号与解决时间，可逐步构建起面向实际故障的参考索引。

**依赖库版本跟踪**：对于项目所依赖的开源库，将其 Release Notes、CVE 公告、兼容性测试报告等链接汇总至 MQSaqL，配合批次时间戳，可在每次版本升级前快速获取相关变更信息，降低升级风险。

**新成员入职引导**：将团队常用的开发规范参考、内部工具文档、核心依赖仓库、云服务状态页等链接集合为一个或多个条目列表，新成员可通过浏览 MQSaqL 快速了解团队所依赖的外部知识体系，缩短上手周期。

## 快速开始

以下操作假定本地已安装 Git 与 Node.js（LTS 版本），若未安装请先前往官网下载。

```bash
# 克隆项目仓库
git clone https://github.com/vhjhbkhggk/mqsaql.git
cd mqsaql

# 安装依赖（用于本地预览与链接格式检查）
npm install

# 生成当前批次的资源列表文档
npm run build -- --batch 100 --output ./dist/resources_100.md
```

执行完成后，在 `./dist/` 目录下会生成包含全部链接的 Markdown 文件。可直接在编辑器内打开查看，或将其作为子模块引入现有文档站点。

若需要重新生成所有批次的汇总索引，可运行：

```bash
npm run build:all -- --output ./dist/README.md
```

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Node.js 18.x 或更高 | 是 | 运行时环境，用于执行构建脚本与格式校验 |
| npm 9.x 或更高 | 是 | 包管理器，用于安装依赖工具 |
| Git 2.30 或更高 | 是 | 用于克隆仓库及提交更新 |
| markdownlint-cli | 否 | 推荐安装，用于本地检查 Markdown 语法规范性 |
| ripgrep 13.0 或更高 | 否 | 可选，用于在大型列表中快速检索关键词 |
| Python 3.9 或更高 | 否 | 仅当需要运行旧版元数据迁移脚本时需要 |

## 贡献指南

**提交新批次链接**：通过 Pull Request 在 `./data/batches/` 目录下新增一个以批次号命名的文本文件（例如 `100.txt`），每行一个完整 URL，保持原始格式不变。PR 描述中请注明该批次的来源或用途简述。

**更新现有条目注释**：若需为已有链接补充注释或修改分类，请直接编辑 `./data/annotations.json` 文件，以链接完整 URL 为键，附加 `comment` 与 `category` 字段，并提交 PR。

**完善格式规范文档**：若发现当前文档导航中的规范存在模糊之处，欢迎在 `docs/format-spec.md` 中补充示例或澄清规则。变更需保持与既有条目的兼容性。

**报告失效链接**：通过 Issue 提交失效链接的完整 URL，并附上可替代的新链接（若有）。维护者会定期批量验证并更新状态标记。

**改进构建脚本**：若对 `src/core/` 下的构建逻辑有优化建议，请先编写对应单元测试，再提交代码变更。所有新功能需附带测试用例。

## 常见问题

**问：为什么所有链接都指向同一个 GitHub 仓库下的 .md 文件？这些文件内容是什么？**

答：MQSaqL 当前版本采用集中式仓库存储所有条目元数据，每个链接对应一个独立的 .md 文件，用于存放该链接的标题、摘录、标签与状态信息。这种设计便于通过 GitHub 原生界面逐条查看与编辑，同时保留完整的文件历史记录。这些 .md 文件本身不存储实际外部内容，仅作为索引载体。

**问：如何快速判断一个链接是否仍然有效？**

答：项目提供可选的健康检查脚本 `npm run check:links`，该脚本会发送 HEAD 请求验证每个 URL 的可达性，并将结果输出到 `./reports/health.json`。当前版本该脚本默认跳过，需手动安装 `node-fetch` 依赖后启用。未来版本计划将此检查集成到 CI 流程中。

**问：生成的 Markdown 文档能否直接用于其他静态站点生成器？**

答：可以。`dist/` 目录下输出的文件为标准 CommonMark 格式，兼容 VuePress、Hugo、MkDocs 等主流工具。若需要调整表格样式或列表缩进，可修改 `src/templates/resource-list.hbs` 模板文件后重新构建。
