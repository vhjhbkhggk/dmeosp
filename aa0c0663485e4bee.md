# ResourceBridge 索引聚合系统

ResourceBridge 是一个面向技术文档检索与外部资源聚合的开源索引系统，专注于将分散于各类代码仓库、技术博客、规范文档中的碎片化知识条目，通过标准化哈希索引与结构化目录树进行集中管理。系统以轻量级 Markdown 文件作为底层存储单元，配合自动化元数据提取脚本，为个人开发者、技术团队以及文档站点维护者提供一套可本地部署、可扩展的资源导航解决方案。

项目核心定位为“外链资源的元数据桥接工具”，不直接托管原始内容，而是通过维护目标链接的哈希指纹与分类标签，构建快速查找与关联推荐的辅助层。适用于需要频繁查阅外部参考材料的技术写作、开源贡献者入职培训、以及多仓库依赖文档的版本对照等场景。

## 功能概览

**哈希索引自动生成**：对每个收录的外部链接计算内容摘要哈希，形成固定长度的唯一标识，便于去重与版本追踪，同时避免因源 URL 变更导致索引失效。

**多层目录归类**：内置三级分类体系，按技术领域、文档类型与时效性对资源进行标签化分组，用户可自定义扩展类别而不破坏核心数据结构。

**Markdown 原生存储**：所有索引记录均以 .md 格式文件保存，字段包含原始链接、收录时间、哈希指纹、简要描述与关联标签，完全兼容主流代码托管平台的在线预览功能。

**批量导入与校验**：提供 Python 脚本支持从 CSV、JSON 或纯文本列表批量导入链接，并自动执行可达性检查与重复项过滤，减少人工维护成本。

**只读镜像同步**：支持将本地索引目录通过 rsync 或 git hook 方式发布为只读静态站点，便于团队内部共享，同时保持源数据与远程仓库的增量同步。

**模糊检索与推荐**：基于文件名的哈希前缀实现快速定位，同时结合标签权重提供关联资源推荐，提升信息发现效率。

## 应用场景

技术文档版本对照：当开源项目维护多版本分支文档时，可使用 ResourceBridge 为每个版本独立创建索引快照，通过哈希比对快速识别外部链接在不同版本间的差异，避免因链接失效导致文档构建失败。

新成员入职知识梳理：团队可将常用技术规范、内部设计文档、第三方 API 手册等外部资源统一收录至索引系统，新成员通过浏览目录树即可掌握知识体系全貌，减少零散查找时间。

离线文档站点构建：对于需要在内网环境部署技术文档站点的团队，可预先通过 ResourceBridge 抓取所需外部链接的元数据，结合静态站点生成器构建只读导航页，确保核心参考资料在隔离网络下仍可被定位。

开源项目 README 资源管理：开源作者可将项目相关的教程、示例代码仓库、社区讨论帖等外部链接统一纳入 ResourceBridge 管理，并在主 README 中仅保留本项目索引链接，降低主文档的维护复杂度。

## 快速开始

```bash
git clone https://github.com/yourorg/resourcebridge.git
cd resourcebridge

pip install -r requirements.txt

python scripts/import_links.py --input resources/initial_links.txt --output index/

python scripts/build_static.py --source index/ --target docs/
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.8 及以上 | 核心脚本运行环境，用于导入、校验与静态生成 |
| git | 2.25 及以上 | 用于克隆仓库及版本管理，支持增量更新 |
| pip | 21.0 及以上 | Python 包管理工具，安装依赖库 |
| rsync（可选） | 3.2 及以上 | 用于同步静态站点到远程服务器 |
| make（可选） | 4.0 及以上 | 支持 Makefile 自动化任务执行 |
| markdownlint-cli（可选） | 0.31 及以上 | 校验索引文件格式规范 |

## 贡献指南

1. 在 GitHub 仓库中提交 Issue 说明拟新增的资源类别或索引条目范围，等待维护者确认需求以避免重复劳动。建议先阅读 docs/index_specification.md 确认索引格式规范。

2. 克隆仓库并创建新的功能分支，分支命名采用 feat/描述 或 fix/描述 格式。所有索引条目需放置在 index/entries/ 下的对应哈希前缀目录中，并确保文件名符合 16 位十六进制哈希规范。

3. 运行 scripts/validate_urls.py 校验新增链接的有效性，并执行 scripts/import_links.py --dry-run 进行模拟导入，确认无格式错误或哈希冲突后再执行正式导入。

4. 更新 docs/ 目录下相关文档，若新增分类需同步修改 index/categories/ 中的 TOML 配置文件，并补全 meta/manifest.json 中的版本号与条目计数。

5. 提交 Pull Request 并附上变更说明，CI 流程将自动执行测试用例与链接校验。合并前至少需一名维护者审阅通过。

## 常见问题

Q: 哈希文件名是如何生成的？如果原始链接发生变动，是否需要重新生成？

A: 哈希基于原始 URL 的标准化字符串（去除末尾斜杠与片段标识）计算 SHA-1 摘要并截取前 16 位。若链接指向的目标内容迁移但 URL 保持不变，则无需重新生成；若 URL 本身变更，则视为新条目，旧条目可标记为废弃状态。

Q: 索引目录增长过大时如何管理查询性能？

A: 系统采用两级前缀目录（前两位哈希）进行分片，单目录下文件数控制在 256 个以内。同时提供 scripts/build_static.py 生成 JSON 索引缓存，供高频只读查询使用。建议每月执行一次链接可达性校验并清理失效条目。

Q: 是否支持私有仓库或内网链接？

A: 本系统仅存储链接字符串本身，不涉及内容抓取，因此理论上支持任何协议的链接。但校验脚本的超时与重试机制默认基于公网环境，若需在内网使用，请修改 scripts/utils/config_loader.py 中的请求参数。
