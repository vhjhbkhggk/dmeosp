# ResourceIndex Core

ResourceIndex Core 是一个面向开发者与技术研究人员的结构化外链与文档资源聚合系统。该项目不提供具体的内容生产功能，而是围绕高质量外部技术资源建立索引、分类与快速导航机制，帮助用户在复杂的技术生态中高效定位所需的参考材料、示例代码、配置模板与调试记录。项目定位为技术资源的“元目录”，适用于个人知识库补充、团队技术文档外部参考整合以及自动化文档流水线的源头数据管理。

目标用户包括：需要频繁查阅外部技术要点的运维工程师、从事基础架构研究的系统开发者、维护大型项目文档体系的技术写作者，以及希望建立可复用资源引用规范的技术管理者。通过集中管理分散于代码仓库、笔记系统与即时通讯记录中的关键链接，ResourceIndex Core 将非结构化的外链数据转化为可检索、可版本化、可审计的结构化资源清单。

## 功能概览

批量资源导入 支持从 Markdown 列表、CSV 及 JSON 格式批量导入 URL 记录，自动解析文件元信息并写入索引数据库。

哈希标识索引 每条资源记录均以内容哈希值作为主键，确保同一 URL 在不同导入批次中自动去重，并保留首次与末次访问时间戳。

分类标签系统 允许为每条链接附加多级标签，支持按项目名称、技术领域、文档类型、重要程度等维度进行细粒度筛选。

状态标记与过滤 内置资源可用性检查机制，可标记链接为有效、失效、需复审或永久归档，并支持按状态过滤输出视图。

只读镜像导出 支持将当前索引导出为静态 HTML 或纯 Markdown 格式的只读镜像，便于内网分发或版本冻结。

访问统计聚合 记录每条资源的本地引用次数与外部访问频次，生成简单的热度排序，辅助判断资源实际使用价值。

命令行交互界面 提供完整的 CLI 工具，支持资源添加、删除、查询、标签修改及批量状态更新，无需依赖图形界面。

## 应用场景

技术文档编写时的外部参考管理 技术作者在编写系统设计文档或运维手册时，需要引用大量外部规格说明、API 参考或社区讨论。ResourceIndex Core 可为每篇文档生成独立的引用清单，并自动检查链接有效性，避免文档发布后出现死链。

团队新人入职环境搭建指引 新成员加入团队时通常需要访问一系列内部工具链、代码仓库模板以及基础环境配置说明。运维负责人可将相关资源预先导入系统，按角色生成定制化的资源列表，大幅缩短新人上手周期。

自动化 CI/CD 流水线中的资源快照 在持续集成流水线中，可将资源索引导出为固定版本的 JSON 文件，与构建产物一同归档。当外部依赖文档发生变更时，可通过对比哈希值快速定位影响范围，辅助回归测试决策。

开源项目外部贡献指引聚合 开源项目维护者可以将贡献指南、代码规范、提交信息格式说明、测试环境搭建教程等外部链接统一收录，减少在 Issue 和 PR 讨论中反复重复相同链接的沟通成本。

## 快速开始

以下命令演示了从克隆仓库到启动本地索引服务的完整流程。

```bash
git clone https://github.com/example/resourceindex-core.git
cd resourceindex-core
pip install -r requirements.txt
python scripts/import_batch.py --input ./data/initial_links.csv --tags "onboarding,reference"
python scripts/serve.py --port 8080 --mirror ./output/mirror
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 及以上 | 核心运行时环境，用于执行导入、查询与导出逻辑 |
| SQLite | 3.35 及以上 | 内置索引数据库引擎，支持 JSON 字段与生成列 |
| Git | 2.30 及以上 | 用于克隆仓库及版本化导出资源的增量提交 |
| Pandas | 1.4.0 及以上 | 仅在批量导入 CSV 或 Excel 格式数据时作为可选加速依赖 |
| PyYAML | 6.0 及以上 | 用于解析配置文件中的标签映射与分类规则 |
| Jinja2 | 3.1 及以上 | 用于将索引数据渲染为静态 HTML 镜像页面 |
| requests | 2.28 及以上 | 用于执行资源可用性检查与状态探测 |
| click | 8.1 及以上 | CLI 命令行界面的基础框架，提供子命令与参数解析 |
| pytest | 7.2 及以上 | 仅开发测试阶段需要，用于运行单元测试与集成测试套件 |

## 贡献指南

1. 阅读 docs/api_design.md 了解核心数据模型与扩展点，确认你的改动方向与现有架构兼容。对于新增功能，建议先在 Issue 中描述设计思路，获得核心维护者反馈后再着手实现。

2. 克隆仓库并在本地创建功能分支，分支命名格式为 feature/简述改动 或 fix/问题编号。确保所有新增代码均包含对应的单元测试，测试覆盖率达到 80% 以上。

3. 运行 pytest 套件确保现有功能未被破坏，同时执行 scripts/cron_check.py 模拟实际运行环境，验证状态检查逻辑无异常。若涉及导出模板修改，需在本地启动 serve.py 预览效果。

4. 提交代码时遵循约定式提交规范，使用 feat:、fix:、docs:、refactor: 等前缀，并在提交信息中明确关联的 Issue 编号。提交前自动执行 pre-commit 钩子进行代码格式检查。

5. 发起 Pull Request 并填写合并请求模板，描述改动背景、实现方案、测试结果以及可能的影响范围。至少需要一位核心维护者 Approve 后方可合并。

## 常见问题

Q: 资源索引数据库是否会随着导入次数增加而无限膨胀？

A: 系统采用哈希去重机制，相同 URL 在多次导入中只会保留一条记录，同时通过定期清理未引用标签和归档失效链接来控制数据库体积。建议每月执行一次 vacuum 命令回收已删除空间。

Q: 如何批量更新已有资源的状态或标签？

A: 使用 cli/commands/status.py 中的 batch-update 子命令，配合 --filter 参数按现有标签或哈希前缀筛选，再通过 --set-tags 或 --set-status 统一修改。该操作支持 dry-run 模式，可预先查看影响范围。

Q: 导出的静态镜像是否可以直接部署到 Nginx 或 GitHub Pages？

A: 可以。export/html_renderer.py 生成的镜像包含完整的 index.html 及 assets 目录，所有链接均为相对路径，可直接放置于任意静态 Web 服务器根目录下。镜像中的外部链接保持原始 URL，不进行重写或代理。
