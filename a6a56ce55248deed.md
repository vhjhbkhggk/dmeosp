# MQSASL Resource Aggregator

MQSASL 是一个面向开发者与运维人员的技术资源外链汇总工具，专注于从分布式版本控制系统中收集、索引和展示与系统架构、中间件配置、安全策略及性能调优相关的技术文档链接。项目本身不存储任何实际内容，仅提供结构化导航与元数据检索能力。

项目目标用户包括后端开发工程师、站点可靠性工程师、DevOps 实践者以及技术决策者。通过将分散在多个仓库中的技术参考文档按主题分类并建立快速访问路径，MQSASL 显著降低了在大型代码库中定位特定技术方案的时间成本，同时通过批次化管理机制确保资源列表的可维护性与可追溯性。

## 功能概览

- **按批次索引的资源导航**：基于 114 个独立批次对 180 个核心资源进行分组管理，当前批次为第 98/114 批，涵盖系统底层与中间件相关文档。

- **基于哈希的持久化引用**：每个资源条目以固定长度哈希值作为唯一标识符，确保在仓库重组或迁移过程中引用链路的稳定性与可复现性。

- **多维度元数据检索**：支持按文档主题、技术栈、适用版本及更新日期进行过滤与排序，满足不同角色对技术文档的差异化查阅需求。

- **轻量化本地部署**：项目完全基于静态 Markdown 构建，无需数据库或额外运行时环境，克隆后即可在任意 HTTP 服务器或本地文件系统中直接访问。

- **自动化资源校验脚本**：内置链接可用性检查工具，可定期扫描资源列表中的外链状态并生成可读性报告，辅助维护者及时更新失效引用。

- **贡献者友好型扩展接口**：通过标准化的资源录入模板与批次分配策略，允许社区成员在无需了解项目整体架构的前提下提交新的资源条目。

- **兼容主流文档管理规范**：资源分类体系遵循技术文档常见布局，与 Confluence、GitBook 及 MkDocs 等平台的目录结构保持语义一致性。

## 应用场景

**微服务架构文档集中管理**：当团队维护数十个微服务仓库时，各仓库的 README 或 docs 目录中分散着大量架构决策记录与配置示例。MQSASL 可将这些分散链接汇总至统一导航页，使新成员能够快速了解整体服务拓扑与关键依赖关系。

**技术选型与评估参考**：在进行中间件选型或框架升级时，工程师需要查阅大量官方文档及社区实践案例。通过 MQSASL 的批次化资源列表，可系统性收集特定主题下的参考链接，形成结构化的评估材料供团队讨论。

**CI/CD 流水线附属文档源**：在自动化构建与部署流程中，可集成 MQSASL 的资源校验脚本，将其作为流水线的一个附属步骤，用于验证所有外部依赖文档的可用性，并在文档链接失效时触发告警，保障运维手册的实时有效性。

**离线文档镜像准备**：对于网络受限的内网开发环境，MQSASL 的资源列表可作为离线文档下载任务的输入清单，配合 wget 或 curl 批量抓取脚本，快速构建本地技术文档镜像站。

## 快速开始

以下操作在 Linux 或 macOS 环境下完成，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/vhjhbkhggk/mqsaql.git
cd mqsaql

# 安装依赖工具（用于链接校验与静态服务）
npm install -g markdown-link-check http-server

# 在当前目录启动静态 HTTP 服务，默认监听 8080 端口
http-server -p 8080

# 如需校验所有外链状态，可执行项目内置脚本
./scripts/check-links.sh ./README.md
```

访问 `http://localhost:8080` 即可在浏览器中浏览当前批次的资源导航页面。若需生成离线 HTML 版本，可使用 `./scripts/build-static.sh` 脚本将 Markdown 转换为独立页面。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Git | 2.20 及以上 | 用于克隆仓库及管理版本更新 |
| Node.js | 14.x 或 16.x LTS | 运行链接校验工具及静态服务器（非运行核心功能必需，建议安装） |
| npm | 6.x 及以上 | 安装 markdown-link-check 及 http-server 等辅助工具 |
| Bash | 4.0 及以上 | 执行项目内置的自动化脚本（Linux/macOS 默认环境） |
| curl | 7.0 及以上 | 用于脚本中执行 HTTP 请求测试链接可用性 |
| grep | 3.0 及以上 | 辅助脚本解析 Markdown 文件中的链接行 |

## 贡献指南

MQSASL 欢迎来自社区的各种贡献，包括但不限于新增资源链接、更新失效地址、优化文档结构以及改进自动化脚本。请按照以下步骤提交您的贡献：

1. 在 GitHub 上 Fork 本仓库至您的个人账户，并克隆到本地开发环境中。确保您的本地分支基于最新的 main 分支创建。

2. 根据您要贡献的内容类型，在对应的目录下进行修改。新增资源条目时，请使用 templates/resource-entry.tmpl 模板填写完整的元数据信息，包括标题、原始链接、所属技术领域及简要描述。对于链接修复，请在提交信息中注明原始链接与更新后链接的对比。

3. 运行项目内置的测试脚本以验证您的修改未引入格式错误或链接语法问题。执行 ./tests/metadata-validate.test.sh 检查元数据完整性，执行 ./scripts/check-links.sh 验证所有外链可访问。

4. 提交 Pull Request 至本仓库的 main 分支。请在 PR 描述中清晰说明本次变更的目的、涉及的资源批次编号以及任何需要维护者特别关注的细节。PR 标题应遵循 [Batch-xxx] Short Description 的格式。

5. 等待维护者审核。审核过程中可能会要求您补充信息或调整格式。合并后，您的贡献将出现在下一版资源索引中，并会在贡献者列表中署名。

## 常见问题

**问：资源列表中的哈希值是如何生成的？它们与原始文档内容有何对应关系？**

答：每个哈希值是该资源条目在入库时根据其原始链接、标题及添加时间戳计算的 SHA-256 前缀，长度为 16 个十六进制字符。该哈希值仅作为内部唯一标识符使用，不与原始文档的内容哈希挂钩。原始文档的实际内容以目标 URL 指向的仓库版本为准。若需追溯某个哈希对应的具体文档上下文，可查看 docs/resources/batch/098.md 文件中该哈希条目旁的备注字段，其中记录了添加时的简要说明。

**问：我发现某个资源链接已经失效，应该如何处理？**

答：请先访问该链接确认是否暂时性不可用。若确认已永久失效，欢迎按照贡献指南提交 Pull Request，在对应的批次详情文件中将该条目标记为 [DEPRECATED] 并提供可替代的链接（若有）。您也可以直接在 Issues 中报告失效链接，维护者会定期处理。项目内置的 check-links.sh 脚本每周自动运行一次，其生成的报告会公开在 docs/operations/maintenance.md 中，供社区参考。

**问：项目是否支持按技术栈或关键词筛选资源？**

答：当前版本通过文档导航表格和批次详情页面提供基础的分类索引。全文检索功能目前依赖于浏览器的页面内查找（Ctrl+F）。计划在后续版本中引入基于 Lunr.js 的静态站点搜索功能，届时将支持按标题、描述及技术标签进行模糊匹配检索。如有紧急的筛选需求，可临时使用 grep 命令在本地 Markdown 文件中进行正则匹配。
