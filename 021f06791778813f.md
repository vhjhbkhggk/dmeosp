# LinkVault Core

LinkVault Core 是一个面向开发者与技术研究人员的结构化外链与资源归档工具。该项目并非传统意义上的网址导航，而是一套基于 Git 仓库管理的技术文献、代码片段、配置模板与运维日志的索引系统。其核心定位在于将分散于 GitHub 议题、技术博客、官方文档及内部知识库中的零散链接，通过固定的仓库目录结构与 Markdown 元数据描述，转化为可追溯、可校验、可批量更新的本地化资源映射表。

项目主要解决个人开发者与小型团队在维护技术选型文档、搭建私有 Helm 仓库索引、管理 Ansible 剧本依赖、或是归档深度学习实验日志时面临的“链接失效、上下文丢失、检索困难”三大问题。LinkVault Core 不提供爬虫与自动抓取功能，而是强调人工审核与 Git 版本管理的结合，确保每一条外链的留存都附带收录原因、标签体系与快照摘要。通过预设的批处理框架（当前迭代至第 35/114 批，共纳入 180 个资源链接），用户可快速建立具备企业级可读性的外链资产库。

## 功能概览

**批次化导入框架** 支持按批次（每批 180 条链接）导入原始 URL 列表，自动根据文件名哈希生成唯一索引 ID，并校验重复条目与格式合规性。

**Markdown 元数据驱动** 每个资源条目均以独立 .md 文件存储，包含 frontmatter 头部（标题、标签、收录日期、来源项目）与正文摘要，便于静态站点生成器或 IDE 插件进行全文检索。

**目录树自动生成** 基于存放路径与文件名前缀，自动生成 ASCII 目录树结构并输出至 README，帮助维护者快速定位某一主题（如 k8s 配置、Python 工具链）下的所有链接。

**依赖环境检测脚本** 内置 bash 与 Python 混合编写的健康检查工具，可批量验证链接的可访问性（HTTP 状态码）与域名过期时间，并将异常条目输出至警告日志。

**标签与全文检索** 利用 grep 与 ripgrep 实现离线关键词检索，无需启动数据库服务即可在毫秒级内从 180 条记录中匹配出相关条目。

**版本化审计追踪** 每次新增或删除链接均需通过 Pull Request 流程合并，仓库的 Commit 历史完整记录了资源的变更原因与操作人，满足内控审计要求。

## 应用场景

**技术选型调研归档** 架构师在评估 Service Mesh 方案时，可将 Istio、Linkerd、Envoy 的官方文档、性能对比博客、GitHub 议题链接统一收录至 LinkVault 的 `meshes/` 目录下，并为每条链接添加“选型依据”与“优缺点摘要”。后续团队评审时，可直接拉取最新批次链接进行集中阅读，避免重复搜索。

**DevOps 剧本依赖管理** 运维工程师维护 Ansible Playbook 时，可将 roles 依赖的 Galaxy 角色地址、第三方 collection 的 Git 仓库、内部私有 PyPI 索引地址纳入 LinkVault。配合健康检查脚本，可在 CI 流水线中每日定时检测依赖源是否可访问，提前发现上游仓库迁移或 TLS 证书变更导致的部署中断。

**深度学习实验日志索引** 算法研究员在多次调参过程中，会将 TensorBoard 链接、W&B 项目地址、数据集下载源（如 Kaggle、HuggingFace）、预训练模型 checkpoint 存储路径写入 LinkVault 的 `experiments/` 子目录。每一轮实验均对应一个批次，方便回溯某一超参数组合下的所有外部参考依据。

**私有知识库外链白名单** 企业合规部门可借助 LinkVault 维护一份“允许访问的技术社区与论坛”列表，将 Stack Overflow、GitHub、GitLab、Docker Hub 等域名统一登记，并关联内部员工手册。当新员工申请网络权限时，安全团队可直接导出一份经过审核的外链清单。

**离线文档站资源映射** 内网文档中心若采用 MkDocs 或 VuePress 构建，LinkVault 可作为其底层数据源，将文档中引用的外部图片、JavaScript 库 CDN 地址、字体文件源统一管理。构建脚本在打包前遍历 LinkVault 记录，自动替换或预下载外部资源，实现内网完全离线化部署。

## 快速开始

以下命令演示如何在本地克隆仓库、安装基础依赖并运行首次批处理导入。

```bash
# 克隆仓库到本地
git clone https://github.com/your-org/linkvault-core.git
cd linkvault-core

# 安装依赖（Python 3.9+ 与 ripgrep）
pip install -r requirements.txt
# macOS 额外需安装: brew install ripgrep
# Ubuntu/Debian: sudo apt install ripgrep

# 运行批次导入脚本，指定第 35 批原始数据文件
python scripts/batch_import.py --batch 35 --input data/raw/batch_35_urls.txt
```

执行完毕后，所有链接将被解析并生成对应的 .md 条目存放于 `archive/batch_35/` 目录下。用户可通过 `scripts/health_check.py --batch 35` 验证链接有效性。

## 安装要求

| 依赖项 | 必需版本 | 说明 |
|--------|----------|------|
| Python | 3.9 及以上 | 用于运行导入、校验与检索脚本，较低版本不支持类型注解语法 |
| ripgrep | 13.0.0 及以上 | 提供高性能纯文本搜索，替代 grep 以加速关键词检索 |
| Git | 2.30 及以上 | 用于版本控制、分支管理与 Commit 签名校验 |
| Bash | 4.0 及以上 | 运行自动化脚本中的 shell 辅助函数，需支持数组与字符串替换 |
| GNU Coreutils | 8.30 及以上 | 提供 date、sort、uniq 等命令，确保日期格式化与去重逻辑正确 |

## 贡献指南

1.  Fork 本仓库至个人命名空间，并克隆到本地。创建新分支时请遵循 `feature/batch-{编号}` 或 `fix/health-check-{简述}` 的命名模式，确保分支名称具备语义化。

2.  新增批次前，先在 `data/raw/` 目录下准备 `batch_{编号}_urls.txt` 文件，每行一个 URL。运行 `python scripts/batch_import.py --batch {编号} --dry-run` 进行预检查，确认无格式错误或重复条目后，再执行正式导入。

3.  对于已归档的 .md 文件，若需修改摘要内容或更新标签，请直接编辑文件并提交 PR。修改时需同步更新文件头部的 `last_modified` 字段为当前 ISO 8601 时间戳，并附上 Commit 信息说明变更原因。

4.  提交 Pull Request 前，请确保所有单元测试通过：在项目根目录执行 `pytest tests/`。若新增了解析器或校验逻辑，需补充对应的测试用例至 `tests/` 下相应模块。

5.  PR 合并前需至少一名维护者审阅，并验证健康检查脚本对本次新增的所有链接均返回 2xx 或 3xx 状态码。若存在不可达链接，需在 PR 描述中注明原因（如临时维护、永久迁移），并在条目 frontmatter 中标记 `status: unreachable`。

## 常见问题

**Q: 批次编号的规则是什么？是否允许跳号或补录？**

批次编号从 1 递增至 114，每批次固定包含 180 条链接。原则上不允许跳号或补录早期批次，以确保归档历史的连续性与审计可追踪性。若因特殊原因需插入历史批次，应创建 `batch_{编号}_supplement` 子目录，并在 README 的“版本历史”章节中单独说明。

**Q: 健康检查脚本如何区分“临时不可达”与“永久失效”？**

脚本采用指数退避重试机制，在首次超时后间隔 5 秒、15 秒、30 秒重试三次。若三次均失败，则进一步检查域名 WHOIS 信息与 TLS 证书有效期。若域名已过期或证书吊销，则判定为“永久失效”；否则标记为“临时不可达”，并在下次定时任务运行时再次检测。用户也可手动运行 `health_check.py --force-recheck` 强制立即重检。

**Q: 是否支持私有 GitLab 或企业 Gitea 实例上的链接？**

目前解析器原生支持 GitHub、GitLab.com 与 Gitee 的 blob 和 raw 链接。对于自托管的 GitLab 或 Gitea 实例，用户可在 `config/settings.yaml` 中新增 `custom_domains` 列表，并实现对应的 `parse_custom_domain` 函数。开发者文档中提供了完整的抽象类与示例代码，便于二次开发。
