# Resource Atlas

Resource Atlas 是一个面向开发者与技术研究人员的结构化外链资源汇总系统。该项目不对资源内容进行二次存储或镜像，而是通过严谨的目录分类与索引机制，将分散于网络各处的高质量技术文档、代码仓库、配置示例与工程笔记聚合为统一的检索入口。项目定位为个人与团队的技术外链收藏与管理工具，适用于需要系统化维护外部学习资料、减少重复搜索成本、以及建立团队共享知识索引的场景。

该项目并非通用的书签管理器，而是针对技术资源特性设计了基于文件命名规范与哈希标识的索引体系，确保每个外链均可被快速定位、归类与版本追溯。通过 Markdown 文件与目录结构的结合，Resource Atlas 使得外部链接如同本地文档一般可被检索、引用与归档。

## 功能概览

- 外链资源索引：以 Markdown 文件为载体，每条资源记录包含原始链接、收录时间、标签与简短摘要，所有条目以哈希值作为文件名确保唯一性。

- 目录分类体系：资源按照技术领域、用途或来源进行多级目录划分，支持用户自定义分类规则与层级深度。

- 批量导入与校验：支持通过脚本批量导入 URL 列表，并在导入过程中自动检测链接可达性与格式合规性，减少无效条目。

- 全文检索支持：所有 Markdown 文件内容可被主流全文搜索引擎索引，支持按文件名、标签、摘要内容进行快速查找。

- 版本关联追踪：通过 Git 进行版本管理，每次增删改操作均有提交记录，便于回溯资源变更历史与审核修改来源。

## 应用场景

1. 个人技术学习路线管理：开发者在学习新框架或语言时，可将查阅过的官方文档、教程文章、示例项目等外链统一收录，按学习阶段分类，后续复习或撰写笔记时可快速调取原始资料。

2. 团队共享知识库构建：技术团队可将日常开发中参考的第三方库文档、故障排查帖子、性能优化案例等资源集中存储于同一仓库，新成员入职时可通过该索引快速了解团队常用技术栈与参考资料。

3. 开源项目依赖文档归档：开源维护者可将项目所依赖的外部规范、协议说明、上游参考实现等链接整理为结构化目录，便于贡献者理解设计决策与合规要求。

4. 技术资讯与周刊素材管理：技术内容创作者可定期收集一周内的技术动态、发布公告、优秀博文，通过该索引进行暂存与分类，为编写技术周刊或汇总报告提供素材源。

5. 离线文档映射辅助：对于需要在内网环境工作的场景，可将外链与内部镜像地址或离线包路径建立映射关系，本索引作为映射查找的前端入口。

## 快速开始

以下命令演示从克隆仓库到启动本地索引服务的完整流程，默认使用 Python 3 内置的 HTTP 服务器进行本地预览。

```bash
git clone https://github.com/resource-atlas/resource-atlas.git
cd resource-atlas
pip install -r requirements.txt
python scripts/build_index.py --source ./links --output ./dist
python -m http.server 8000 --directory ./dist
```

执行上述命令后，打开浏览器访问 http://localhost:8000 即可查看索引首页。若只需命令行检索，可运行 `python scripts/search.py --keyword "docker"` 进行快速查找。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.8 及以上 | 用于运行索引构建脚本与本地服务器 |
| Git | 2.25 及以上 | 用于克隆仓库及版本管理操作 |
| Markdown 解析器 | commonmark 0.9+ | 用于渲染索引页面，脚本会自动安装 |
| 磁盘空间 | 100 MB 可用 | 实际占用取决于资源数量，每条记录约 1-5 KB |
| 网络访问 | 可选 | 仅首次克隆与后续更新时需访问远程仓库，日常使用无需联网 |
| 操作系统 | Linux / macOS / Windows WSL2 | 脚本兼容主流 Unix-like 环境，Windows 推荐使用 WSL |
| 内存 | 512 MB 及以上 | 构建索引时内存占用与条目数线性相关，180 条资源约需 50 MB |
| 文件系统 | 支持大小写敏感 | 推荐 ext4 或 NTFS，文件名哈希使用大小写混合 |

## 贡献指南

贡献者需遵循以下流程以确保索引的一致性与可维护性。所有提交须附带清晰的变更说明，并经过本地构建验证。

1. 克隆仓库并创建特性分支：从 main 分支拉取最新代码，使用 `git checkout -b feature/your-feature-name` 创建新分支，避免直接在主干提交。

2. 新增或修改资源文件：在 links 目录下选择合适的二级分类子目录，使用 `scripts/import_batch.py` 工具导入新 URL，或手动创建 Markdown 文件并确保文件名符合哈希命名规范。每个文件必须包含原始链接、收录日期与至少一个分类标签。

3. 运行本地校验与构建：执行 `python scripts/validate_links.py` 检查新增链接的有效性，修复所有报错后运行 `python scripts/build_index.py` 确保索引页面可正常生成。

4. 提交并推送变更：使用清晰的提交信息，格式为 `[类别] 简短描述`，例如 `[database] 新增 PostgreSQL 15 官方文档链接`。推送至远程分支后，通过 Pull Request 提交审核。

5. 参与代码审查：PR 描述中需说明变更动机与影响范围，至少一名维护者审核通过后方可合并。合并后 CI 将自动重新构建并更新在线索引页面。

## 常见问题

Q: 资源文件中的原始链接发生变更或失效，项目如何维护？

A: 项目本身不负责链接内容的可用性保证，但提供了 `scripts/validate_links.py` 脚本用于定期检查。用户可手动运行该脚本生成失效报告，并提交 PR 更新或移除无效条目。建议每季度运行一次全面校验。

Q: 能否在资源文件中添加个人笔记或摘要内容？

A: 可以。每个 Markdown 文件除必须包含原始 URL 外，允许在下方自由添加摘要、标签、阅读状态等自定义字段。但需注意保持格式简洁，避免过长的个人笔记影响索引页面的渲染性能。

Q: 如何将本项目的索引数据迁移到其他系统或格式？

A: 所有资源数据均以纯 Markdown 格式存储，文件内元数据采用 YAML Front Matter 或键值对形式，可被多种静态站点生成器解析。项目也提供了 `scripts/export_csv.py`（未在默认安装中包含，可自行编写）用于导出为 CSV 或 JSON 格式，便于导入 Notion、Airtable 等其他平台。
