# RepoIndex

RepoIndex 是一个面向技术团队与独立开发者的外链资源索引与结构化文档聚合工具。该项目通过将分散于多个仓库、Issue、PR 及外部文档中的参考链接统一收录，并以标准化 Markdown 形式提供浏览与检索入口，帮助开发者快速定位特定批次的技术资料、历史记录与参考实现。

RepoIndex 不提供新的知识内容，而是作为知识索引层，将原始数据链接组织为可维护、可审阅、可版本化的目录体系。其典型使用对象包括技术文档维护者、项目依赖审计人员以及需要回溯特定技术决策背景的研发工程师。

## 功能概览

**结构化资源收录**：按照项目批次与来源对原始链接进行归类，每个链接均保留原始 URL 及其上下文属性，确保可追溯性。

**多维度元数据标注**：为每条资源链接附加文件名称、提交哈希特征、所属模块等辅助信息，便于后续过滤与检索。

**自动化目录生成**：基于链接集合自动生成项目结构树与资源列表，减少人工维护成本，保持文档与实际仓库状态同步。

**版本化快照支持**：每个收录批次对应一个独立目录快照，允许开发者在不同版本之间对比资源增减与变更情况。

**全文检索接口**：提供简单的 grep 风格命令行查询工具，支持按文件名、哈希前缀或自定义标签快速定位目标链接。

**导出与集成能力**：支持将资源列表导出为 CSV、JSON 或纯文本格式，便于导入外部文档系统或自动化脚本处理。

## 应用场景

**技术文档资源归档**：当团队维护大量外部参考链接（如 RFC 文档、博客文章、规范草案、依赖库发布说明）时，可使用 RepoIndex 按批次导入并生成可浏览的索引页，避免链接散落在邮件或聊天记录中。

**依赖变更影响分析**：在版本升级或安全补丁应用前，通过查询 RepoIndex 中记录的关联链接，快速了解本次变更所涉及的参考文档、测试用例链接或相关 Issue 讨论，从而评估影响面。

**开源合规审计辅助**：对于需要梳理第三方代码引用来源的项目，RepoIndex 可将所有外部链接统一登记，配合人工审核标记合规状态，生成审计所需的引用清单。

**新成员技术背景衔接**：为新加入团队的工程师提供按主题或批次组织的链接列表，使其能够系统性地阅读项目所依赖的技术资料与历史决策记录，缩短上手周期。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆仓库
git clone https://github.com/your-org/RepoIndex.git
cd RepoIndex

# 安装依赖（Python 3.9+ 与 pip）
pip install -r requirements.txt

# 运行索引构建脚本，生成当前批次的资源列表与目录树
python build_index.py --batch 110-114 --output docs/
```

执行完毕后，docs/ 目录下将生成 index.md 文件，即包含所有资源链接的主索引文档。

| Python 3.9 或更高版本 | 是 | 用于运行索引构建、校验与导出脚本 |
| Git 2.25 或更高版本 | 是 | 用于克隆仓库及版本管理操作 |
| GNU Make 3.81 或更高版本 | 推荐 | 用于执行自动化构建流程与测试套件 |

| PyYAML 6.0 或更高版本 | 是 | 用于解析配置文件与元数据描述文件 |
| requests 2.28 或更高版本 | 可选 | 用于验证外部链接的可达性（需网络访问） |

## 项目结构

```
RepoIndex/
├── build_index.py            # 主构建脚本，读取配置并生成索引文档
├── config.yaml               # 全局配置文件，定义批次、分类与输出格式
├── requirements.txt          # Python 依赖列表
├── Makefile                  # 自动化任务入口（构建、测试、清理）
├── docs/                     # 生成的文档输出目录
│   ├── index.md              # 主索引页，包含全部资源列表与目录树
│   ├── user-guide.md         # 用户手册
│   └── maintainer-guide.md   # 维护者指南
├── src/                      # 核心源码目录
│   ├── parser.py             # 链接解析与校验模块
│   ├── tree_builder.py       # 目录树生成器
│   └── exporter.py           # CSV/JSON 导出实现
├── tests/                    # 单元测试与集成测试
│   ├── test_parser.py
│   └── test_tree_builder.py
├── scripts/                  # 辅助运维脚本
│   ├── link_checker.sh       # 外部链接可达性检测
│   └── batch_import.py       # 批量导入新批次链接
└── .github/                  # GitHub 工作流配置
    └── workflows/
        └── build.yml         # CI 自动构建流水线
```

## 贡献指南

1. 克隆项目仓库并在本地环境中安装所有必需依赖（参见安装要求表格），确保 Python 版本不低于 3.9。

2. 在 config.yaml 中按照现有格式添加新批次定义，包括批次编号、描述以及对应的原始链接列表文件路径。链接列表文件需按行存放 URL，每行一条。

3. 运行 make test 执行全部单元测试与集成测试，确认新增配置未引入解析错误或格式异常。若测试失败，请根据输出信息修复配置文件或相关脚本。

4. 提交 Pull Request 至主仓库的 develop 分支，并在 PR 描述中注明新增批次的编号、链接总数以及是否执行过本地构建验证。PR 至少需要一位项目维护者审阅通过后方可合并。

## 常见问题

**Q：索引构建脚本报错提示 "YAML 解析失败"，应如何排查？**

A：此错误通常由 config.yaml 中的缩进不一致或使用了非法字符引起。建议使用支持 YAML 语法高亮的编辑器（如 VS Code 配合 YAML 插件）检查配置文件，并确保所有字段缩进使用空格而非制表符。也可以运行 yamllint config.yaml 进行自动检测。

**Q：如何验证外部链接是否仍然有效？**

A：项目提供了 scripts/link_checker.sh 脚本，该脚本会遍历当前批次所有链接并发送 HEAD 请求以检查可访问性。执行时需要确保网络可达，且部分网站可能因防火墙或速率限制导致误报，建议在非高峰时段运行并适当调整超时参数。

**Q：索引页中的目录树是自动生成的吗？**

A：是的。目录树由 src/tree_builder.py 根据当前仓库实际目录结构动态生成，并附带从 .treeignore 文件中读取的排除规则。如果希望添加或隐藏特定目录，请编辑 .treeignore 文件后重新运行 make build。
