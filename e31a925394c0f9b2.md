# MQSASL Resource Index

MQSASL Resource Index 是一个面向开发者和技术研究人员的结构化外链资源汇总工具。该项目不对资源内容进行二次加工或镜像存储，而是提供一套基于哈希标识的索引机制，用于对分散在多个数据源中的技术文档、配置文件、日志片段与调试快照进行统一引用与快速定位。

该项目适用于需要长期维护大量外部引用链接的团队，尤其适合在代码审查、故障复盘、依赖审计等场景中作为引用锚点使用。MQSASL 不限制资源类型，可涵盖 Markdown 技术笔记、配置模板、SQL 脚本、调试输出文件等各类文本形态的外部资产。

## 功能概览

**哈希键值索引**：每个资源条目均以 16 进制哈希字符串作为唯一标识，便于在版本控制系统和自动化脚本中追踪资源变动。

**扁平化目录存储**：所有引用记录统一存放于 main 分支下的固定目录，避免深层嵌套带来的路径解析开销。

**资源状态标记**：每个索引条目可附带状态标签，用于标识资源的可用性、版本兼容性或审核状态。

**批量导入支持**：提供脚本接口用于将批量生成的哈希引用列表快速导入索引库。

**跨项目复用**：同一套索引文件可被多个上游项目通过 submodule 或符号链接方式引用。

**只读镜像友好**：目录结构设计兼顾静态站点生成器与 CDN 分发需求，便于构建只读镜像。

**审计日志关联**：每条索引记录可关联外部审计系统的单号或工单编号。

**模糊查找兼容**：支持基于哈希前缀的部分匹配查询，适用于无法获取完整哈希的应急场景。

## 应用场景

在微服务架构的依赖清单管理中使用 MQSASL 索引文件记录每个第三方库的配置文件哈希，当上游仓库发布新版本时，仅需更新对应的索引条目即可同步所有下游项目的引用记录。

在安全事件响应流程中，安全团队可将受影响的配置文件或日志片段上传至内部存储，然后将生成的哈希链接提交至 MQSASL 索引库，便于开发团队快速定位问题资产。

在技术文档写作中，作者可将长篇代码示例或复杂 SQL 查询拆分为独立资源文件，使用 MQSASL 索引替代文档正文中的大段代码块，降低文档维护成本并提升可读性。

在 CI/CD 流水线中，构建脚本可在产物生成后自动将构建日志、测试报告和打包配置的哈希引用追加到索引文件中，形成可追溯的构建资产清单。

## 快速开始

以下步骤演示如何获取 MQSASL 资源索引库的本地副本并启动基础服务。

```bash
git clone https://github.com/vhjhbkhggk/mqsaql.git
cd mqsaql
pip install -r requirements.txt
python scripts/validate_index.py --check-all
```

上述命令执行完毕后，可通过本地 HTTP 服务查看索引内容：

```bash
python -m http.server 8080 --directory ./docs
```

访问 http://localhost:8080 即可浏览当前索引库中的所有资源条目。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.9 及以上 | 是 | 用于运行索引验证与导入脚本 |
| Git 2.25 及以上 | 是 | 用于克隆仓库及管理索引文件的版本历史 |
| pip 20.0 及以上 | 是 | 安装 Python 依赖包所需 |
| markdown2 2.4.0 | 是 | 用于将索引描述转换为 HTML 预览 |
| pyyaml 6.0 | 是 | 解析索引元数据配置文件 |
| pytest 7.0 | 否 | 仅在运行单元测试时需要 |
| pre-commit 3.0 | 否 | 用于本地提交前的格式校验 |
| http.server 模块 | 否 | Python 内置模块，用于快速预览 |
| curl 或 wget | 否 | 用于远程获取资源文件进行验证 |

## 贡献指南

新增索引条目时，请先在 scripts 目录下运行 validate_index.py 脚本检查待添加的哈希是否已存在，避免重复引用。确认无冲突后，将新的 .md 文件放入 index 目录，并在本地提交前执行 pre-commit 钩子进行格式校验。

更新已有索引条目时，若仅修改描述文本或状态标签，无需变动文件名。若指向的外部资源内容发生变更且哈希值随之改变，则需要删除旧文件并以新哈希命名重新创建，同时更新相关引用文档中的链接。

提交变更前，请确保所有新增或修改的 .md 文件均包含必要的元数据头，至少包括资源类型、原始来源和最后验证时间三个字段。元数据格式参考 docs/reference/schema.json 中的定义。

若发现索引库中的某个链接已失效，请在 GitHub Issues 中提交一个问题报告，并在标题中注明失效条目的哈希前缀。管理员会定期处理失效条目，处理方式包括标记为废弃或删除。

对于批量导入需求，请参考 docs/usage/batch_import.md 中的说明，准备符合格式要求的 CSV 或 JSON 文件，然后运行 import_batch.py 脚本完成导入。

## 常见问题

问：索引文件中的哈希值是如何生成的？

索引文件名的 16 进制字符串取自原始资源内容的 SHA-256 哈希值前 16 位。这种截断方式在降低文件名的视觉复杂度的同时，仍能保证在 180 个条目的规模下发生碰撞的概率低于可接受阈值。完整的 SHA-256 哈希值记录在每个 .md 文件内部的元数据字段中，供需要完整校验的场景使用。

问：如何验证索引库中的所有链接均有效？

项目提供了 validate_index.py 脚本，该脚本会遍历 index 目录下的所有 .md 文件，提取其中记录的原始资源 URL，并发送 HEAD 请求检查可达性。校验结果会生成一份 HTML 报告，存放在 docs/reports/ 目录下。用户也可通过 GitHub Actions 的定时任务自动执行校验，并在校验失败时收到邮件通知。

问：索引库是否支持私有化部署？

支持。MQSASL 索引库本身仅存储哈希引用和元数据描述，不存储实际资源内容，因此可以完整克隆到内网环境使用。管理员只需将 config.yaml 中的 upstream_base 字段修改为内网存储服务的地址，即可将索引指向内部资源。
