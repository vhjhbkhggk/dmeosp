# MetaDocs Collective

MetaDocs Collective 是一个面向开发者、技术团队与技术内容消费者的结构化技术资源导航与元数据汇总项目。项目以 Git 仓库形式组织，收录并分类大量外部技术文档链接、参考手册、社区讨论快照与代码示例索引，旨在解决技术资料分散、版本混乱与检索效率低下的问题。

项目目标用户包括：需要快速查阅特定技术栈文档的一线研发人员、负责技术选型与知识库建设的架构师、以及希望系统化整理学习路径的在校学生或转岗工程师。项目本身不托管文档原内容，而是通过索引与分类机制，将分散于 GitHub 议题、技术博客、官方手册与社区讨论中的高质量资源聚合为可检索、可追溯的引用集合。

## 功能概览

- **结构化资源索引**：所有外部链接按主题、来源与用途划分为多级目录，支持人工筛选与脚本批量检索。

- **哈希锚定引用**：每个资源条目附带唯一内容哈希标识，便于在团队内部文档或自动化工具中直接引用。

- **元数据标注系统**：每条资源均标注文档类型（手册/博客/规范/讨论）、目标读者与预估阅读时长。

- **版本快照关联**：对于外部链接，记录采集时的提交哈希或发布日期，以降低内容变更导致的引用失效风险。

- **全文检索支持**：项目内置轻量级元数据 JSON 索引，可配合 grep、jq 或 Elasticsearch 进行关键词与标签检索。

- **批量导入与校验**：提供脚本工具，支持从 CSV、Markdown 列表或书签 HTML 批量导入新链接，并自动校验 URL 可达性。

- **离线浏览模式**：通过构建脚本可将索引资源生成为静态 HTML 站点，供内部网络或本地环境使用。

## 应用场景

- **团队技术周报素材整理**：技术负责人每周浏览项目新增索引，快速筛选与本周开发任务相关的文档链接，直接附入周报，减少重复搜索时间。

- **新成员技术 onboarding**：为新人提供按技术栈（如 Go、Rust、前端工程化）预过滤的资源列表，配合阅读顺序建议，使其在两周内建立系统认知。

- **技术选型调研辅助**：架构师在进行中间件或框架选型时，通过项目中的对比类文章索引与官方手册快照，快速汇总各方案的优劣势与社区活跃度。

- **离线知识库镜像构建**：在无法直连外网的生产环境或实验室网络中，利用项目的离线生成功能，将已索引的公开文档转化为内部可用的静态站点。

## 快速开始

以下命令可在本地克隆项目并启动基础服务。

```bash
# 克隆仓库
git clone https://github.com/your-org/metadocs-collective.git
cd metadocs-collective

# 安装依赖（Python 3.9+ 与 pip）
pip install -r requirements.txt

# 构建本地索引并启动预览服务
python build_index.py --input ./data/raw_links.json --output ./dist/index.html
python -m http.server 8080 --directory ./dist
```

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Python | 3.9 或更高 | 用于运行构建脚本与校验工具 |
| Git | 2.30 或更高 | 用于克隆仓库及提交更新 |
| curl | 7.68 或更高 | 用于外部链接可达性检查（可选） |
| jq | 1.6 或更高 | 用于处理 JSON 元数据（可选） |
| Node.js | 18.x 或更高 | 用于前端预览站点构建（可选） |
| make | 3.81 或更高 | 用于执行自动化任务脚本 |
| sqlite3 | 3.31 或更高 | 用于本地元数据缓存（可选） |

## 贡献指南

1. 在 GitHub 上 fork 本仓库，并克隆至本地。创建新分支时使用 `feat/` 或 `fix/` 前缀，例如 `feat/add-docs-about-rust`。

2. 新增外部链接请编辑 `data/raw_links.json`，严格按照已有 schema 填写 `url`、`category`、`title` 与 `description` 字段。若不确定分类，可使用 `uncategorized` 并提交 issue 讨论。

3. 修改元数据后，运行 `make test` 执行本地校验，包括 JSON 格式检查、必填字段完整性与 URL 格式合法性。所有测试通过后方可提交。

4. 提交 commit 时遵循约定式提交规范（Conventional Commits），例如 `feat: add 20 links for database tuning` 或 `docs: update contributing guide`。

5. 发起 Pull Request 至 `main` 分支。PR 描述中请说明新增或修改的资源范围，以及是否涉及死链修复或分类调整。至少一位 maintainer 审核通过后合并。

## 常见问题

**问：如何报告某个外部链接已经失效或内容与描述不符？**

答：请在本仓库的 Issues 中选择 "Broken Link" 模板，提供失效链接的完整 URL 与采集时的哈希值。若确认链接内容已迁移，可附上新 URL，维护团队将在下一个周期更新索引并重新生成预览站点。

**问：项目是否接受自动化的批量 PR？**

答：接受。但批量 PR 必须附带运行日志或校验报告，说明所有新增链接均通过可达性检查，且分类与标签符合现有 schema。一次 PR 建议不超过 50 条新链接，以便审阅者逐项核对。

**问：如何离线使用整个索引？**

答：执行 `make build` 会在 `dist/` 下生成完整的静态 HTML 站点，包含所有链接的分类列表与搜索框。将该目录整个拷贝至内网服务器或本地文件系统，通过浏览器打开 `index.html` 即可离线检索。
