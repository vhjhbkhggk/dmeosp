# ResourceBridge MQL

ResourceBridge MQL 是一个面向技术文档整理与开源知识库构建的静态资源导航与元数据索引项目。该项目定位于为开发者、技术作者与开源社区维护者提供一套可本地化部署、低依赖、高扩展性的外链资源汇总方案，特别适用于需要长期维护大量技术参考链接、代码示例索引或社区文档引用的场景。

项目核心解决以下问题：技术社区中优质外链资源分散、链接失效后难以追溯、跨项目引用缺乏统一入口、以及文档中散落链接难以进行批量校验与版本管理。ResourceBridge MQL 通过将外链资源以结构化 Markdown 文件的形式组织在单一仓库中，配合轻量级索引机制，使维护者能够以 Git 工作流管理全部外链资源，并支持通过脚本进行可用性检测与格式规范校验。

本项目第 21/114 批资源整合共包含 180 个独立条目，均以散列命名存储于 mqsaql 目录下，涵盖技术文档、代码片段、配置模板与架构说明等多种类型。

## 功能概览

**结构化外链仓库**：以纯 Markdown 文件形式集中存储全部外链资源，每个文件包含元数据头与正文内容，便于人工阅读与程序解析。

**基于文件名的散列索引**：每个资源文件以 16 位十六进制散列值命名，避免命名冲突，同时支持通过辅助索引表进行主题检索。

**零运行时依赖**：项目本身无需数据库或后端服务，所有内容均为静态文件，可直接通过任意 HTTP 服务器或 GitHub Pages 发布。

**批量链接校验脚本**：提供基于 Python 的链接可用性检查工具，支持并发检测与结果输出，帮助维护者快速定位失效链接。

**Markdown 格式强制规范**：内置基于 markdownlint 的规则集，确保所有文档标题层级、代码块标注与表格格式保持一致。

**版本追踪与审计**：所有资源变更均通过 Git 提交记录保留，支持按时间、作者与内容类型进行回溯审计。

**自定义元数据扩展**：支持在 Markdown 文件顶部 YAML Front Matter 中定义 tags、category、source 与 status 字段，便于分类筛选。

## 应用场景

技术博客的外部引用归档：技术作者在撰写多篇文章时可统一将外部参考链接归档至 ResourceBridge MQL，文章正文仅保留本地引用标识，后续链接变更仅需在仓库中统一修改，避免逐篇文章手动更新。

开源项目文档的外链治理：开源项目维护者可将项目 README、贡献指南与 API 文档中的所有外链迁移至本仓库，通过版本管理追踪每一次外链变更，并在 CI 流程中集成链接校验步骤，确保文档质量。

团队内部知识库的链接中心：企业技术团队可将常用的内部系统地址、云服务控制台、运维手册与设计文档链接统一存放于本仓库，配合目录分类与标签筛选，形成团队共用的资源门户。

技术培训与课程材料管理：培训机构或大学课程讲师可将课程涉及的所有阅读材料、在线工具与示例代码仓库链接集中管理，每期课程可基于特定提交版本生成稳定的资源列表，避免授课过程中外部资源变动影响教学。

## 快速开始

以下操作步骤适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash 执行。

```
git clone https://github.com/vhjhbkhggk/mqsaql.git
cd mqsaql
pip install -r requirements.txt
python scripts/check_links.py --target ./main --output report.json
```

如需在本地启动静态预览服务，可使用 Python 内置模块：

```
python -m http.server 8000
```

访问 http://localhost:8000 即可浏览 main 目录下的全部资源索引页面。

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Python | 3.8 及以上 | 用于运行链接校验与索引生成脚本 |
| Git | 2.25 及以上 | 用于克隆仓库与提交变更 |
| pip | 20.0 及以上 | 安装 Python 依赖包 |
| markdownlint-cli | 0.31.0 及以上 | 用于本地 Markdown 格式检查（可选） |
| curl | 7.68 及以上 | 用于脚本中的 HTTP 请求测试（可选） |
| rsync | 3.1.0 及以上 | 用于增量同步资源文件到发布目录（可选） |
| jq | 1.5 及以上 | 用于 JSON 报告解析（可选） |
| shellcheck | 0.7.0 及以上 | 用于辅助脚本的 Shell 语法检查（可选） |

## 贡献指南

提交新资源条目：在 main 目录下新建一个以 16 位十六进制散列值命名的 .md 文件，文件顶部必须包含 YAML Front Matter，其中至少定义 title、category 与 source 字段。正文部分使用 Markdown 书写，首段应为资源摘要说明，后续可包含详细描述与使用示例。

更新已有资源：若某一条目的外链地址或描述信息发生变更，直接修改对应的 .md 文件，并在提交信息中注明变更原因。如需标记某资源为已失效，将 status 字段更新为 deprecated 并在正文中说明替代方案。

提交变更流程：Fork 本仓库，在个人分支上完成修改后发起 Pull Request。PR 标题需按 [资源类型] 简短描述 的格式填写，正文中说明变更背景与自检结果。PR 通过 CI 检查后方可合并。

报告链接失效：请在 GitHub Issues 中选择 Broken Link Report 模板，填写失效链接所在文件的散列名、原始目标 URL 以及当前返回的 HTTP 状态码。维护者将在 5 个工作日内处理。

本地测试建议：提交前请运行 scripts/check_links.py 对新增或修改的文件进行链接可达性验证，并执行 scripts/format_lint.sh 确保 Markdown 格式符合项目规范。

## 常见问题

问：资源文件为何使用散列值命名而非可读性名称？

答：散列命名可避免多人同时编辑时产生文件名冲突，且与 Git 的版本管理机制兼容性更好。同时，散列值文件名的不可读性促使维护者依赖索引表与元数据进行检索，从而保证索引机制的持续维护与更新。用户可通过 indexes/by_category.json 或 by_tag.json 快速定位目标资源。

问：如何处理外部链接已经失效的情况？

答：项目提供了基于 Python 的链接校验脚本 scripts/check_links.py，可定期扫描所有资源文件中的外链并生成状态报告。若发现失效链接，建议首先尝试通过 Internet Archive 或其它缓存服务寻找可替代地址；若原内容已彻底移除，则将对应资源文件的 status 字段更新为 deprecated，并在正文末尾添加失效备注。不建议直接删除资源文件，以保留历史审计信息。

问：本项目能否与静态站点生成器配合使用？

答：可以。ResourceBridge MQL 的设计目标之一就是作为静态站点的数据源层。用户可通过 scripts/generate_index.py 生成 JSON 索引，再将其导入 Hugo、Jekyll 或 VuePress 等生成器，利用索引数据渲染资源导航页面。项目本身不限定前端框架，索引数据格式为标准的 JSON 结构，可被任意语言解析。
