# ResourceBridge 索引网关

ResourceBridge 是一个面向开发者与研究人员的高密度技术资源索引网关，专注于对分散在代码托管平台、技术文档站点与知识库中的外部链接进行结构化采集、分类归档与状态监控。本项目不存储任何实际内容，仅提供元数据映射与引用关系管理，适用于需要长期维护大量外部参考链接的技术团队、开源项目维护者以及个人知识管理场景。

ResourceBridge 解决的核心问题是：当技术参考资料分散在数百个独立页面中时，如何通过统一的索引层实现快速检索、失效检测与批量导出。项目内置轻量级爬取调度器与链接状态检查器，能够定期验证每个收录 URL 的可达性，并自动标记异常条目。同时提供标签系统与全文搜索支持，帮助用户在海量链接中精确锁定所需资源。

## 功能概览

批量链接收录：支持从 Markdown 列表、CSV 文件或标准输入流中批量导入外部链接，自动提取页面标题与摘要元数据。

状态监控与告警：周期性检查所有收录链接的 HTTP 状态码与响应时间，对 4xx/5xx 错误、超时及证书异常生成预警报告。

多维度标签分类：允许用户为每条链接自定义标签组合，支持按项目名称、技术领域、文档类型、语言版本等维度进行筛选与聚合。

全文检索与过滤：基于倒排索引提供对链接标题、描述、标签及页面摘要的全文搜索，支持布尔运算符与通配符查询。

数据导入导出：支持将索引数据导出为 JSON、CSV 或 HTML 书签格式，亦支持从浏览器书签文件或 Pinboard 备份中导入现有收藏。

链接关系图谱：可视化展示链接之间的引用关系与聚类结构，帮助用户发现主题相近的资源群落。

RESTful API 接口：提供完整的 CRUD 操作接口与查询端点，便于与其他自动化工具或持续集成流水线集成。

## 应用场景

技术文档库维护：开源项目维护者可以使用 ResourceBridge 管理项目文档中引用的所有外部参考链接，包括依赖库主页、API 参考手册、教程文章与社区讨论帖。当外部资源迁移或下线时，系统能够及时发出通知，避免文档中出现死链。

学术研究参考文献管理：研究人员在收集论文、预印本、数据集与工具仓库时，可将所有相关 URL 导入 ResourceBridge，按照研究方向、实验编号或出版年份进行标注。内置的链接状态检查功能确保在论文发表或项目交付时所有引用仍然有效。

团队知识库聚合：企业技术团队可以将内部 Wiki、Confluence 空间、GitLab 仓库与外部技术博客中分散的链接统一纳入 ResourceBridge 索引，通过标签和搜索功能实现跨平台知识的快速定位，减少重复劳动与信息孤岛。

个人书签系统替代：长期积累的书签文件往往难以维护和检索。ResourceBridge 提供了比传统浏览器书签管理器更强大的分类、搜索和健康检查能力，并且数据完全自主托管，支持多终端同步。

## 快速开始

以下命令序列演示了从克隆仓库到启动服务的完整流程。

```bash
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge
pip install -r requirements.txt
python -m resourcebridge.server --port 8080 --datastore ./data
```

完成上述步骤后，在浏览器中访问 http://localhost:8080 即可进入管理界面。首次启动时系统会在 ./data 目录下创建默认的 SQLite 数据库与索引文件。若需使用 PostgreSQL 作为后端存储，请参考 docs/configuration.md 中的高级配置说明。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.9 或更高版本 | 是 | 核心运行时环境，建议使用 3.11+ 以获得性能优化 |
| SQLite 3.35+ | 是 | 默认元数据存储引擎，支持 JSON 函数与全文搜索扩展 |
| requests 2.28+ | 是 | 用于发出 HTTP 请求以验证链接状态与获取页面信息 |
| beautifulsoup4 4.11+ | 是 | 解析 HTML 页面以提取标题、描述与正文摘要片段 |
| jinja2 3.1+ | 是 | 渲染 Web 管理界面的模板引擎 |
| pytest 7.0+ | 否 | 仅在运行单元测试与集成测试时需要 |
| psycopg2 2.9+ | 否 | 当选择 PostgreSQL 作为后端数据库时需要安装 |
| redis 4.5+ | 否 | 用于分布式部署时的缓存与任务队列后端 |
| lxml 4.9+ | 否 | 作为 beautifulsoup4 的备选解析器，可提升解析速度 |

## 贡献指南

本项目的开发遵循常规的 GitHub Flow 协作模式。欢迎提交问题报告、功能请求与代码变更。在提交 Pull Request 之前，请确保遵循以下流程：

1. 在 Issue 列表中搜索是否存在相同或相似的问题与功能请求，避免重复劳动。若未找到相关议题，请新建一个 Issue 描述您计划解决的问题或新增的功能，并说明实现思路。

2. 从 main 分支创建新的功能分支，分支命名建议采用 feat/ 或 fix/ 前缀，后跟简要描述。例如 feat/add-batch-import 或 fix/validator-timeout。所有开发工作应在该分支上进行。

3. 编写代码时请遵循 PEP 8 风格规范，并为新增的功能或修复编写对应的单元测试。测试覆盖率不应低于现有水平。提交前运行 pytest 确保所有测试通过。

4. 提交 Pull Request 时请在描述中关联对应的 Issue 编号，并详细说明变更内容、测试结果以及任何可能影响现有行为的改动。PR 至少需要一名项目维护者的审阅与批准。

5. 文档更新：任何新增功能或 API 变更都应同步更新 docs/ 目录下的相关文档。若变更涉及配置选项，请同时在 .env.example 中添加说明注释。

## 常见问题

Q: 如何迁移现有的浏览器书签或 Pocket 收藏夹到 ResourceBridge？

A: 项目提供了多种导入途径。对于浏览器书签，可以将书签导出为 HTML 文件，然后使用 scripts/import_bookmarks.py 脚本进行转换导入。对于 Pocket 等在线服务，建议先导出为 CSV 格式，再通过 Web 界面中的批量导入功能上传。具体操作步骤请参考 docs/quickstart.md 中的导入章节。

Q: 链接状态检查的频率和并发数如何调整？是否会对外部站点造成压力？

A: 状态检查的调度间隔通过环境变量 CHECK_INTERVAL_HOURS 设置，默认值为 24 小时。并发请求数由 MAX_CONCURRENT_CHECKS 控制，默认值为 5，以避免对目标服务器造成过大负载。此外，系统会遵守 robots.txt 中的 Crawl-delay 指令，并自动跳过已标记为永久失效的链接。生产环境中建议根据实际网络条件和目标站点的响应能力调整这些参数。

Q: 是否支持多用户登录和权限控制？

A: 当前版本专注于单用户模式，所有数据共享同一索引空间。多用户支持已在 2.0 版本的路线图中，届时将引入基于角色的访问控制（RBAC）与用户隔离的数据视图。对于临时性的团队协作需求，建议通过部署多个独立实例或使用反向代理进行路径隔离。
