# TechRef Navigator

TechRef Navigator 是一个面向技术决策者与开发团队的技术资源外链汇总与快速导航工具。项目定位于将碎片化的技术文档、深度分析文章、架构方案与故障排查记录按主题归集，形成可检索、可追溯的外部知识库索引，帮助用户在技术选型、问题排查与团队培训场景中快速找到高质量的第一手资料。

项目本身不存储任何实际内容，仅维护外部链接的元数据与分类标签。通过 Markdown 文件清单与自动化校验脚本，确保所有外链的可用性与分类一致性。当前批次为第 16/114 批，共计收录 180 个经过初步筛选的技术文档外链，内容覆盖后端架构、数据库调优、监控可观测性、安全实践与工程效率等领域。

## 功能概览

批量外链导入与去重 支持从 CSV、Markdown 列表或纯文本批量导入 URL，自动识别重复条目并生成冲突报告。

分类标签与全文检索 每条外链可绑定多个层级标签（如 数据库 / MySQL / 性能优化），支持按标签组合过滤与标题内容模糊匹配。

链接存活检测 内置基于 Headless 浏览器的定时检测任务，自动标记响应超时或状态码异常的外链，并输出失效报表。

元数据增强与注解 允许为每条外链添加自定义摘要、关键段落引用与阅读优先级标记，便于团队协作审阅。

只读镜像发布 生成静态只读的导航站点，可将外链清单发布为内部文档站点，支持暗色模式与移动端适配。

权限分级管理 基于角色控制外链的新增、编辑、归档与删除权限，支持对接 LDAP 或 OAuth2 提供方。

导出与集成 支持将当前批次的外链数据导出为 JSON、YAML 或 SQLite 文件，便于集成到 Confluence、Notion 或内部 Wiki 中。

## 应用场景

技术选型调研 架构师在评估消息队列或数据库方案时，通过本项目集中查阅来自 GitHub 的深度评测笔记与性能对比数据，避免在搜索引擎中重复过滤低质量信息。

故障复盘与知识沉淀 运维团队在处理线上事故后，将相关的根因分析文章、Hotfix 提交记录与监控截图链接归集到同一批次，形成可追溯的事故档案。

新人培训路线图 团队负责人按技术领域筛选一批入门与进阶文章，生成专属阅读清单，新员工可在独立环境中按序学习，同时通过注解功能记录疑问与心得。

技术雷达维护 技术委员会定期扫描外链清单，标注 采纳 / 试验 / 观望 状态，以此驱动季度技术雷达的更新，确保技术决策有据可查。

文档站外链治理 文档团队利用本项目的失效检测功能，定期扫描产品文档中的所有外部引用，批量替换或移除已失效的链接，提升文档健康度。

## 快速开始

以下命令演示如何从 GitHub 克隆项目、安装依赖并启动本地导航服务。

```bash
git clone https://github.com/techref-navigator/techref-navigator.git
cd techref-navigator
pip install -r requirements.txt
python serve.py --port 8080 --batch 16
```

执行上述命令后，本地导航服务将运行在 8080 端口，默认加载当前批次（第 16 批）的外链清单。访问 http://localhost:8080 即可看到只读导航界面。如需导入新的外链列表，可使用 `python importer.py --input urls.txt --batch 16` 命令。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|----------|----------|------|
| Python | 3.9 及以上 | 核心服务运行环境，建议使用 3.11 LTS |
| pip | 22.0 及以上 | Python 包管理工具，用于安装依赖库 |
| SQLite | 3.35 及以上 | 默认元数据存储引擎，无需额外配置 |
| Git | 2.30 及以上 | 用于克隆仓库及版本管理 |
| curl | 7.68 及以上 | 链接存活检测依赖的命令行工具 |
| Node.js | 18.x LTS | 仅在前端构建时需要，运行时不强制 |
| npm | 8.x 及以上 | 前端资源打包工具，仅构建时依赖 |
| Docker | 20.10 及以上 | 可选，用于容器化部署 |
| 内存 | 512 MB 及以上 | 服务运行最低内存要求，推荐 1 GB |

## 贡献指南

提交外链新增建议 通过 Issue 模板提交新增外链的请求，需提供 URL、标题、推荐标签与 50 字以内的摘要说明。维护者将在 48 小时内审核并合并至待导入队列。

参与链接存活检测优化 如果发现检测脚本存在误报或超时参数不合理，请提交带有详细日志的 Bug 报告。欢迎提交针对不同站点类型（GitHub、技术博客、官方文档）的自定义检测策略。

完善文档与翻译 文档仓库支持中英文双语，欢迎补充使用案例、更正过时的截图或增加 FAQ 条目。翻译需保持技术术语的一致性。

开发前端主题扩展 前端采用极简 CSS 变量设计，欢迎提交新的配色主题或布局改进。需确保移动端响应式不被打断。

本地测试与提交规范 所有提交需通过 `pytest tests/` 单元测试，并遵循 Conventional Commits 格式（feat / fix / docs / chore）。提交前请运行 `scripts/pre-commit.sh` 进行基础检查。

## 常见问题

Q: 项目是否存储外链内容的副本或快照？
A: 不存储。项目仅维护 URL 及其元数据（标题、标签、摘要）。所有内容均指向原始来源，用户点击后直接跳转至外部站点。建议用户自行评估外部内容的可靠性与时效性。

Q: 链接存活检测的频率与超时时间是多少？
A: 默认每 24 小时执行一次全量检测，超时阈值为 10 秒。检测结果会缓存 12 小时，避免重复请求。对于 GitHub 原始内容链接，检测器会自动跟随重定向并校验最终状态码。

Q: 如何迁移到 PostgreSQL 或其他关系型数据库？
A: 核心抽象了存储接口，默认使用 SQLite 便于单机部署。如需迁移，可继承 `storage.BaseStorage` 类并实现 `insert`、`query`、`update` 等方法，然后在 `config/settings.yaml` 中切换 `storage_backend` 配置项。项目未内置其他数据库驱动，需要自行安装对应适配器。
